# 房产专家Agent 小程序 - 设计文档

## 1. 项目概述

### 1.1 产品定位
一个面向购房者/投资者的微信小程序，输入小区名称和房屋信息后，自动生成专业的房产价值分析报告。核心价值是将房产专家的经验法则内化成可量化的智能分析引擎。

### 1.2 目标用户
- 刚需购房者：判断目标房源是否值得买
- 房产投资者：快速评估投资价值
- 卖房者：了解合理售价区间

### 1.3 设计原则
- **0服务器成本**：优先前端计算，必要时使用微信云开发免费额度
- **高性能**：缓存策略，减少API调用
- **可扩展**：规则引擎可配置，支持不断迭代专家逻辑
- **审核安全**：不爬取第三方数据，不展示房源，纯工具定位

## 2. 系统架构

### 2.1 总体架构

```
用户输入层
    ↓
数据采集层（地图API + 用户手动补充）
    ↓
专家规则引擎（核心估值模型）
    ↓
结果展示层（分析报告 + 估值区间）
```

### 2.2 技术选型

| 层级 | 技术方案 | 说明 |
|------|----------|------|
| 前端 | 微信小程序原生 + TypeScript | 已有的技术栈 |
| 后端 | 微信云开发（云函数+云数据库） | 免费额度，无需服务器 |
| 地理编码 | 天地图API | 完全免费 |
| POI搜索 | 高德地图API | 个人5000次/月，配合缓存 |
| 缓存 | 云数据库 + 本地Storage | 双层缓存，降低API消耗 |

### 2.3 数据流

```
用户输入小区名
    ↓
本地缓存命中？→ 是 → 直接返回分析结果
    ↓ 否
云数据库缓存命中？→ 是 → 返回结果 + 本地缓存
    ↓ 否
调用天地图地理编码 → 获取经纬度
    ↓
调用高德POI搜索 → 获取周边配套/硬伤
    ↓
专家规则引擎计算 → 生成分析报告
    ↓
存入云数据库缓存 + 本地缓存
    ↓
展示给用户
```

## 4. 专家规则引擎设计

### 4.1 估值公式

```
最终估值 = 基础单价 × 面积 × 区域系数 × 配套系数 × 硬伤系数 × 楼栋系数
```

### 4.2 区域系数

| 等级 | 定义 | 系数 |
|------|------|------|
| 核心核心区 | CBD、顶级商圈 | 1.15 ~ 1.25 |
| 核心区 | 主城区、成熟商圈 | 1.05 ~ 1.15 |
| 普通区 | 主城区边缘、新城中心 | 1.00 |
| 非核心区 | 郊区、工业园周边 | 0.85 ~ 0.95 |

**判定规则**：
- 通过行政区划 + 商圈关键词匹配
- 内置重点城市的商圈等级数据库
- 未知区域默认普通区（系数1.0）

### 4.3 配套系数

| 维度 | 权重 | 评分规则 |
|------|------|----------|
| 交通 | 25% | 地铁距离、公交数量 |
| 教育 | 20% | 学校数量、学校等级 |
| 商业 | 15% | 商场数量、餐饮密度 |
| 医疗 | 10% | 三甲医院、社区医院 |
| 自然 | 10% | 公园、水域 |

**配套系数 = 1 + (综合得分 / 100) × 0.3**

### 4.4 硬伤系数（负向修正）

| 硬伤类型 | 判定条件 | 系数 |
|----------|----------|------|
| 高架/主干道噪音 | 距离<50m | 0.75 ~ 0.85 |
| 公墓/殡仪馆 | 距离<1km | 0.65 ~ 0.80 |
| 变电站/高压线 | 距离<300m | 0.80 ~ 0.90 |
| 垃圾处理站 | 距离<500m | 0.75 ~ 0.85 |
| 污水处理厂 | 距离<1km | 0.70 ~ 0.85 |
| 机场航线 | 航线正下方 | 0.85 ~ 0.90 |

**判定方法**：
- 搜索周边POI，匹配关键词
- 计算直线距离
- 多硬伤叠加（累乘，最多扣到0.5）

### 4.5 楼栋系数

| 因素 | 修正范围 | 说明 |
|------|----------|------|
| 楼层 | 0.90 ~ 1.05 | 中间层高，顶/底层低 |
| 朝向 | 0.95 ~ 1.10 | 南北通透>南>东南>东西>北 |
| 楼栋位置 | 0.85 ~ 1.10 | 小区中间>靠边>临路>临垃圾站 |

**楼层系数计算规则**（总楼层N）：
- 1层：0.90
- 2~3层：0.95
- 中间1/3楼层：1.05
- 次顶层：1.03
- 顶层：0.92

## 7. 成本控制策略

### 7.1 API 消耗估算（日均100次查询）

| API | 单次消耗 | 月消耗 | 免费额度 | 超额风险 |
|-----|----------|--------|----------|----------|
| 天地图地理编码 | 1次 | 3000次 | 3万次/月 | 低 |
| 高德POI搜索 | 5次/查询 | 15000次 | 5000次/月 | 高 |

### 7.2 缓存策略

1. **本地缓存**：查询过的小区结果存本地Storage，7天有效期
2. **云数据库缓存**：热门小区存云数据库，所有用户共享
3. **缓存命中率目标**：>80%（即80%的查询不消耗API）

### 7.3 降级策略

- 高德API用完时，自动降级为天地图POI搜索（精度略低）
- 所有API都用完时，提示用户手动输入配套信息

## 10. 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 高德API额度不足 | 高 | 功能降级 | 缓存+天地图备选+引导用户手动输入 |
| 小程序审核不通过 | 中 | 无法上线 | 选对类目、加免责声明、避免交易词汇 |
| 数据不准确 | 中 | 用户不信任 | 透明化计算过程、标注数据来源、加置信度 |
| 用户输入负担重 | 高 | 留存低 | 尽可能自动获取、分步输入、智能默认值 |
